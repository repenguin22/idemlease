# Design

This document explains the architecture and the reasoning behind the
decisions that shaped idemlease. The normative source is the
requirements contract in [docs/REQUIREMENTS.md](docs/REQUIREMENTS.md)
(Japanese, v3.4; the working name `idemtrail` there reads as
`idemlease`) — when this document and the contract disagree, the
contract wins.

## Architecture: a state machine core with an HTTP skin

```
                 ┌──────────────────────────────────────────┐
 client ──HTTP──▶│ httpidem (net/http)                      │
                 │  key grammar · fingerprint · capture ·   │
                 │  replay policy · KeyScope · RFC 9457     │
                 └───────────────┬──────────────────────────┘
                                 │ Begin / Finish (opaque key, fingerprint, payload)
                 ┌───────────────▼──────────────────────────┐
                 │ idemlease core (stdlib only, no net/http)│
                 │  (none) ─Reserve→ reserved ─Complete→    │
                 │     ▲          │                completed│
                 │     └──(lease expiry / Release)──┘       │
                 └───────────────┬──────────────────────────┘
                                 │ Store interface (Reserve/Complete/Release/Get)
                     memstore · redistore · (pgstore, v1.1) · yours
```

The core treats keys, fingerprints, and payloads as opaque values and
exposes exactly two entry points:

- `Begin` — one atomic `Reserve` plus the decision table: no valid
  record → `Proceed` (with a fresh lease token), valid reserved →
  `RejectInFlight`, valid completed → `Replay` or
  `RejectFingerprintMismatch` by fingerprint.
- `Finish` — `Persist` → `Complete`, `Discard` → `Release`, with
  `ErrTokenMismatch`/`ErrNotFound` normalized to `leaseLost=true`.

**Why split, and why this vocabulary (decision 12).** Everything
HTTP-shaped — header grammar, fingerprint algorithm, response capture,
status mapping — lives in `httpidem`, so framework adapters and a
future gRPC interceptor share one state machine and one store
ecosystem. The core keeps the five-word vocabulary
(`Begin`/`Finish` + `Reserve`/`Complete`/`Release`) rather than
collapsing to a two-word Set/Verify style: in-flight exclusivity, the
token CAS, and explicit release are the load-bearing concepts, and
hiding them from the interface would push that state back into every
integration. The split is package-level within one module — `net/http`
is stdlib, so a separate module would add release overhead for nothing.

**The middleware is deliberately thin (§12-13).** It is composed only
of the core `Begin`/`Finish` and exported parts: `KeyFromHeader`,
`Fingerprint`, `Recorder`, `StoredResponse` serialization. A CI canary
(`docs/prototypes/errtrailadapter`) rebuilds an adapter from public API
only, so adapter-implementability cannot silently regress.

## The lease model

A reservation is a *lease*, not a lock: it expires. Lease expiry is the
**only** reclamation path — the core never garbage-collects, and a
caller that gets `Proceed` but never calls `Finish` (a caller bug)
holds the key hostage exactly until the lease runs out (409 until then,
re-executable after). This makes crash behavior a first-class,
predictable state instead of an operational surprise.

Every reservation carries a random token (crypto/rand, 128 bits),
generated by `Begin` — never by the store. `Complete` and `Release` are
compare-and-set on that token, which is what makes lease expiry safe:
if execution A outlives its lease and B re-reserves the key, A's late
`Finish` fails the CAS and is normalized to `leaseLost` — A's client
still gets A's response (the work happened), but only B's result is
stored and replayed. Store implementations that generated or rewrote
tokens would silently break this, so the conformance suite asserts
verbatim token persistence.

## The Store contract

Four methods, three sentinels, and two rules that carry all the safety:

1. **Reserve is a single atomic claim.** An expired record must be
   overwritten as atomically as a first claim — a GET-then-SET
   implementation has a window where two writers both see "expired" and
   both win. `idemleasetest` hammers this exact window with 100
   concurrent reserves over a just-expired record.
2. **Complete/Release are token CAS.** `ErrTokenMismatch` when the
   record is held by someone else, `ErrNotFound` when there is nothing
   to transition; callers treat both as "lease lost".

Expiry is *logical*: a reserved record past its lease deadline and a
completed record past its record TTL are absent, whatever bytes may
still exist. memstore checks timestamps lazily; redistore binds the
record's lifetime to Redis TTLs (relative `PEXPIRE`, so app/Redis clock
skew cannot corrupt semantics) and gets the atomic-overwrite property
for free — expired keys simply do not exist. Store bugs are surfaced,
not absorbed: `Begin` reports a contract violation (an `existing`
record that is nil or malformed) as an error rather than guessing.

## HTTP decisions worth explaining

**Key grammar (decision 10).** Real clients send raw strings; the
draft's mandatory RFC 8941 quoting is where it diverged most from
practice. So the raw form is canonical (any non-control bytes, 1–255)
and quoted strings are decoded for compatibility, with the documented
trade-off that a raw key starting with `"` needs quoting. The NUL byte
is rejected by the grammar, which is exactly why `KeyScope` can use it
as a collision-free separator (`scope + "\x00" + key`).

**Fingerprint.** `SHA-256(method + "\n" + escapedPath + "?" + rawQuery
+ "\n" + body)`. The query is always included with an unconditional
`?`, keeping the encoding prefix-unambiguous. Host and headers are
excluded on purpose: proxies and gateways rewrite them, and a retry
through a different edge node must still fingerprint identically.

**In-flight conflicts are 409 regardless of fingerprint (decision 7).**
While a key is reserved we cannot know whether the stored execution
will match — "wait" (409 + Retry-After) is the only honest answer.
After completion the answer is knowable, so a mismatch becomes "fix
your request" (422).

**429 is discarded by default (decision 3).** Persisting a 429 would
replay it after the rate limit lifts — forever. 5xx and panics are
discarded for the same reason (transient failures must stay
retryable); other 4xx are persisted so retries of a genuinely bad
request are cheap and consistent.

**Failure modes are per-operation (§4.6).** `FailOpen` applies to
`Begin` only: it trades duplicate-protection for availability, and that
trade only exists before the handler ran. After the handler ran, the
captured response is always returned (the work is done); a failed
persist triggers a best-effort release and an explicit warning that
idempotency is temporarily weakened. `Finish` runs on a
`context.WithoutCancel` so a client disconnect cannot lose the
completed work's record.

**Response capture.** The `Recorder` is writer-agnostic so non-net/http
adapters reuse it; the middleware's `http.ResponseWriter` wrapper is a
thin skin. `Flush`/`Hijack` abandon the capture — streamed responses
are not replayable — and pass through via `http.ResponseController`
(Unwrap-chain aware, `ErrNotSupported` when the underlying writer lacks
the capability). The stored payload is a versioned binary encoding
(version byte + uvarint frames), so the format can evolve without
corrupting old records.

## Testing strategy

Three public, stdlib-only conformance suites do the heavy lifting:

- `idemleasetest.RunStoreTests` — the §3.2 store semantics (atomic
  reserve, expiry, token CAS, verbatim persistence)
- `idemleasetest.RunStateMachineTests` — `Begin`/`Finish` end to end
  over any store
- `httpidemtest.RunHTTPTests` — the client-observable HTTP scenarios,
  including fault injection and a *real* lease-expiry-vs-survivor race

Every store — including the in-repo test double — runs the same suites,
against Redis and Valkey in CI. Expiry scenarios wait real time with
short TTLs and never assert inside a short-lease window, so they are
deterministic on slow runners and honest against out-of-process stores.
Two CI invariants are checked mechanically on every push: the core
never (transitively) imports `net/http`, and the root module's build
graph stays stdlib-only.

## Non-goals and their boundaries

- **Exactly-once execution.** Out of scope by definition; the honest
  contract is at-most-one *lease holder*. The v1.1 `pgstore` +
  `CompleteTx` will close the strongest remaining gap by joining the
  business transaction atomically (design gate: a complete
  success/failure matrix before implementation).
- **Client-side key management**, cross-service delivery guarantees,
  fasthttp-family frameworks.
- **Per-route `Require` configuration** — apply the middleware
  per-route instead; one global knob keeps the configuration surface
  honest.
